# 发布模型（开源边界）

本文明确三件事：哪些内容应该公开、哪些应该留在本地、为什么这样做。

## 1）公开仓库根目录

公开发布时，只把 `GitHub上线` 作为 GitHub 仓库根目录。

原因：
- `GitHub上线` 是整理后的分发包（脚本、文档、合规说明）。
- `源码工程` 是本地开发工作区，可能包含内部历史、实验内容和非分发资产。

## 2）应该发布的内容

建议保留：
- `win/`
- `wsl/`
- `common/`
- `docs/`
- `.github/`
- `README.md`、`README.zh-CN.md`
- `LICENSE`、`SECURITY.md`、`THIRD_PARTY_NOTICES.md`、`CONTRIBUTING.md`、`.gitignore`
- `.gitmodules`

## 3）不应发布的内容

不要公开本地运行态和私有工作区产物：
- `~/.codex`
- `~/.agents`
- 任何真实 `.env` 密钥和 token 文件
- `源码工程` 这类本地开发目录及无关私有目录

## 4）为什么 external 子模块保持透明

`external/*` 保持透明是有意的合规设计：
- 便于追溯上游来源与许可证。
- 便于下游用户做安全审计和版本审计。
- 隐藏或改名来源并不能消除许可义务，反而提高合规风险。

## 5）如果你想让公开仓更轻

低风险做法：
- 保留子模块，但在文档中声明它们是可选集成。
- 发布时不初始化子模块；首次使用时由 bootstrap 自动尝试初始化。
- 持续维护第三方声明文件。

高风险做法（不建议）：
- 只截取第三方文件片段而不完整保留归属/许可上下文。

## 6）与发布检查清单联动

公开 push 前至少确认：
1. 秘钥/路径扫描通过。
2. shell 与 PowerShell 解析检查通过。
3. 文档与插件元数据占位符状态正确。
4. 最终 push 只包含 `GitHub上线` 的内容。
